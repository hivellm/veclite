# Benchmarks & Server Parity (gate G1)

This document records the benchmark methodology, the pinned reference hardware
profile (resolves PRD **OQ-1**), the current smoke numbers, and the
server-parity result that together close **gate G1** (SPEC-015 §4–5, NFR-01..04).

## Reference hardware profile (OQ-1)

Two classes are pinned so results are reproducible and comparable across runs:

| Class | Pinned profile |
|---|---|
| **Desktop/laptop** | AMD Ryzen 9 7950X3D (16C/32T), 128 GB DDR5, Windows 10 Pro, Rust 1.96 (release). This is the current dev reference; the numbers below were measured here. |
| **Cloud** | AWS `c7a.4xlarge` (AMD EPYC 9R14, 16 vCPU, 32 GB) — general-purpose compute, no accelerators, matching VecLite's CPU-only design. The nightly full-size benches run here. |

The reference profile is recorded with every published result (CPU model, RAM,
OS, Rust version), per TST-040. Absolute targets (below) are defined against
these profiles; the ±20 % CI fence (TST-041) compares each PR to the previous
run on the **same** runner, not to these absolutes.

## Methodology

- **Benches** — Criterion, in `crates/veclite/benches/veclite_bench.rs`
  (`cargo bench -p veclite`). Corpora are generated by a fixed-seed splitmix64
  (no `rand` dependency), so runs are deterministic. The smoke benches run at a
  modest scale (2 000 × 512 for search, 1 000 × 512 for build, 500 for batch);
  the reference targets are measured at 1M × 512 SQ-8 on the pinned runners.
- **Parity** — `crates/veclite/tests/parity.rs`, gated on `VECLITE_PARITY_URL`
  (a no-op when unset, so a normal `cargo test` never needs the server). It
  loads an identical clustered 512-d corpus into VecLite and a pinned
  `hivehub/vectorizer:3.5.0` container via raw BYO `/insert_vectors`, then
  compares top-10 per query.

```bash
docker run -d --name vlp -p 15002:15002 hivehub/vectorizer:3.5.0
PW=$(docker exec vlp cat /data/.root_credentials | sed -n 's/^password=//p')
VECLITE_PARITY_URL=http://localhost:15002 VECLITE_PARITY_PASSWORD="$PW" \
  cargo test --test parity -- --nocapture
```

## Current results

### Benches (desktop reference, smoke scale)

| Bench | Scale | Median |
|---|---|---|
| `search/top10` | 2 000 × 512, cosine | **≈ 0.92 ms** |
| `index_build` | 1 000 × 512, cosine | ≈ 810 ms |
| `batch_insert` | 500 × 512, cosine | ≈ 238 ms |

Search p50 is well inside the **< 3 ms** target at this scale; the 1M × 512 SQ-8
reference target is measured on the pinned runners (tracked with the full
nightly bench).

### Server parity (TST-030, gate G1)

Against `hivehub/vectorizer:3.5.0`, identical clustered 512-d corpus
(1 000 vectors, 50 queries), cosine:

> **mean top-10 overlap = 0.9920 ≥ 0.99** ✓

The comparison has real discriminating power: a metric mismatch scores ≈ 0.57
and loosely-clustered (ambiguous) data ≈ 0.97 — only correct, structurally clear
data clears the 0.99 floor.

**Finding (recorded for CORE-003 port-back):** the server's `optimized_hnsw`
hardcodes `DistCosine`, so its ANN ranks by cosine regardless of the
collection's configured metric. VecLite's index is metric-correct
(Cosine→DistCosine, Euclidean→DistL2), so parity is asserted on the cosine path
the server actually implements.
