# VecLite C ABI — prebuilt libraries

VecLite ships its C ABI (SPEC-008) as **prebuilt, checksummed shared and static
libraries** for every supported platform, attached to each GitHub release. You
do **not** need a Rust toolchain to use them — download, link, run.

Direct C/C++ integrators use these; so do the Go and C# bindings, which load the
same C ABI.

## What ships (per platform)

Each release attaches one `veclite-ffi-<target>.tar.gz` per target below,
containing:

| File | What |
|---|---|
| `libveclite.so` / `libveclite.dylib` / `veclite.dll` | shared library |
| `libveclite.a` / `veclite.lib` | static library |
| `veclite.dll.lib` (Windows only) | import library for `veclite.dll` |
| `veclite.h` | the C header (generated by cbindgen) |
| `LICENSE`, `SHA256SUMS` | license + per-file checksums |

A top-level `SHA256SUMS.txt` over all the archives is attached alongside them.

## Supported targets

| OS | Architectures | Notes |
|---|---|---|
| Linux (glibc) | x86_64, aarch64 | `manylinux`-compatible |
| Linux (musl) | x86_64, aarch64 | static-friendly (Alpine, etc.) |
| macOS | x86_64, arm64, **universal2** | `universal2` is a fat `.dylib`/`.a` |
| Windows (MSVC) | x86_64, aarch64 | `veclite.dll` + `veclite.dll.lib` |

The shared libraries embed the shipped name: `SONAME=libveclite.so` (Linux),
`install_name=@rpath/libveclite.dylib` (macOS), and the Windows import library
references `veclite.dll` — so `-lveclite` resolves the runtime dependency
correctly on every platform.

## Download and verify

```bash
# pick your target, e.g. x86_64-unknown-linux-gnu
curl -LO https://github.com/hivellm/veclite/releases/download/vX.Y.Z/veclite-ffi-x86_64-unknown-linux-gnu.tar.gz
tar xzf veclite-ffi-x86_64-unknown-linux-gnu.tar.gz
cd veclite-ffi-x86_64-unknown-linux-gnu
sha256sum -c SHA256SUMS      # verify integrity
```

## Link and run

The header is the single include; the ABI is additive-only within a major
version, so gate loaders on `vl_abi_version()`.

### Linux / macOS (gcc or clang)

```bash
cc app.c -I. -L. -lveclite -Wl,-rpath,'$ORIGIN' -o app     # Linux
cc app.c -I. -L. -lveclite -Wl,-rpath,@loader_path -o app  # macOS
./app
```

Or link the **static** library (no runtime dependency):

```bash
cc app.c -I. libveclite.a -lpthread -ldl -lm -o app        # Linux
cc app.c -I. libveclite.a -o app                           # macOS
```

### Windows (MSVC)

```bat
cl /I. app.c veclite.dll.lib          :: dynamic — ship veclite.dll next to app.exe
cl /I. app.c veclite.lib              :: static
```

### Minimal program

```c
#include "veclite.h"
#include <stdio.h>

int main(void) {
    vl_db *db = NULL;
    if (vl_open_memory(&db) != VL_OK) return 1;
    printf("veclite %s (abi %u, format %u)\n",
           vl_version(), vl_abi_version(), vl_format_version());
    vl_db_close(db);
    return 0;
}
```

`ffi/smoke.c` + `ffi/smoke.sh` in the repo are exactly this link-and-run check,
run against every shipped bundle in CI (the no-toolchain gate, REL-020).

## Building the libraries yourself

Not required (use the release artifacts), but if you must:

```bash
cargo build -p veclite-ffi --release          # target/release/veclite_ffi.{so,dylib,dll,a}
cbindgen --config crates/veclite-ffi/cbindgen.toml --crate veclite-ffi --output veclite.h
```

The release workflow renames the Rust `veclite_ffi.*` output to the shipped
`veclite.*` names and (on Windows) regenerates the import library — see
`.github/workflows/veclite-release.yml`.
