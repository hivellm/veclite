# 06 — SDK Bindings

## Nature of the SDKs: native bindings, not clients

Vectorizer's existing SDKs (`sdks/{typescript,python,rust,go,csharp}`) are **network clients** — MessagePack-over-TCP (port 15503) with REST fallback (15002). VecLite SDKs are the opposite: they **embed the compiled Rust core** in the host process. No sockets, no serialization protocol, no connection pool. The API shape intentionally mirrors the Vectorizer SDKs where concepts overlap, so graduating to the server later is a familiar-feeling swap.

```
                Rust core (crates/veclite)
                     │
   ┌─────────┬───────┼────────┬──────────────┐
   ▼         ▼       ▼        ▼              ▼
veclite-py veclite-node  veclite-ffi   veclite-wasm
 (PyO3)     (napi-rs)    (C ABI .h)   (wasm-bindgen)
   │         │             │    │           │
   ▼         ▼             ▼    ▼           ▼
 Python    Node.js        Go   C#/.NET   Browser/Edge/
 wheels    prebuilds    (cgo) (P/Invoke)  Deno/Bun
```

Direct-binding tier (PyO3/napi-rs) for the two highest-volume ecosystems; C-ABI tier for everything else. WASM is its own target with a reduced storage profile.

## Packaging (the SQLite bar: install must never require a Rust toolchain)

| Ecosystem | Package | Mechanism | Prebuilt matrix |
|---|---|---|---|
| Rust | `veclite` (crates.io) | source crate | n/a |
| Python ≥3.9 | `veclite` (PyPI) | PyO3 + maturin, abi3 wheels | manylinux/musllinux/macOS/Windows × x64/arm64 |
| Node ≥18 | `veclite` (npm) | napi-rs, `optionalDependencies` per-platform packages (`@veclite/darwin-arm64`, …) | same matrix |
| Go | `github.com/hivellm/veclite-go` | cgo + bundled static lib per platform | same matrix |
| C# / .NET ≥8 | `VecLite` (NuGet) | P/Invoke + `runtimes/<rid>/native/` | same matrix |
| Browser/Deno/Bun/Edge | `@veclite/wasm` (npm) | wasm-bindgen + JS glue | wasm32 |
| C/C++ & others | GitHub release artifacts | `libveclite.{a,so,dylib}` + `veclite.h` | same matrix |

Heavy ONNX embeddings ship separately so base installs stay lean: `pip install veclite[onnx]`, npm `@veclite/onnx`, NuGet `VecLite.Onnx`, Go build tag `veclite_onnx`.

## C ABI (`veclite-ffi`) — the contract for Go/C#/community bindings

Design rules:

1. **Handle-based**: opaque `vl_db*`, `vl_collection*`; every function returns `int32` status (0 = OK, negative = error code mapped 1:1 from `VecLiteError` variants); results via out-params.
2. **Thread-safe** by construction (core is `Send + Sync`); any thread may call any handle.
3. **No callbacks in v1**; no panics across the boundary (`catch_unwind` at every entry, panic → `VL_ERR_INTERNAL`).
4. **Memory**: library-allocated buffers are freed only by `vl_free_*`; strings UTF-8; vectors as `(const float*, len)`; payloads/filters cross as JSON or MessagePack bytes (caller picks the codec flag — MessagePack reuses the exact wire structs Vectorizer SDKs already have for payload types).
5. **Versioned**: `vl_version()`, `vl_format_version()`; header generated by `cbindgen` in CI.

```c
int32_t vl_open(const char* path, const vl_open_opts* opts, vl_db** out);
int32_t vl_collection_create(vl_db*, const char* name, const vl_coll_opts*, vl_collection** out);
int32_t vl_upsert_batch(vl_collection*, const vl_point* points, size_t n);
int32_t vl_search(vl_collection*, const float* vec, size_t dim, uint32_t k,
                  const char* filter_json, vl_hits** out);
int32_t vl_search_text(vl_collection*, const char* query, uint32_t k, vl_hits** out);
void    vl_hits_free(vl_hits*);
int32_t vl_db_close(vl_db*);
const char* vl_last_error_message(void);   // thread-local detail string
```

## Per-language API sketches

### Python (PyO3)

```python
import veclite

db = veclite.open("app.veclite")                       # or veclite.memory()
docs = db.create_collection("docs", dimension=384, metric="cosine")
docs.upsert("id-1", vector, payload={"lang": "en"})
docs.upsert_batch(points)                               # list[tuple] | numpy 2-D array
hits = docs.search(vector, limit=10,
                   filter={"must": [{"key": "lang", "match": "en"}]})
notes = db.create_collection("notes", auto_embed="bm25", dimension=512)
notes.upsert_text("id", "text…", {"src": "readme"})
hits = notes.search_text("query", limit=5)
```

- **NumPy zero-copy**: `search(np.ndarray)` borrows the buffer; `upsert_batch` accepts a `(n, dim) float32` array without copying per-row. This is the headline perf feature of the Python binding.
- GIL released around every core call (`py.allow_threads`).
- Context-manager support (`with veclite.open(...) as db:`); `db.close()` idempotent.
- Optional `veclite.aio` asyncio facade (thread-pool executor) — sync is the primary API.

### Node.js (napi-rs)

```ts
import { open } from "veclite";

const db = await open("app.veclite");                  // async by default (libuv threadpool)
const docs = await db.createCollection("docs", { dimension: 384, metric: "cosine" });
await docs.upsertBatch(points);                        // Float32Array zero-copy
const hits = await docs.search(query, { limit: 10, filter: { must: [...] } });
const hitsSync = docs.searchSync(query, { limit: 10 }); // sync variants for CLI/scripts
```

- All ops async via napi `AsyncTask` (never block the event loop); `*Sync` twins provided.
- `Float32Array`/`TypedArray` cross zero-copy.

### Go (cgo over `veclite-ffi`)

```go
db, err := veclite.Open("app.veclite", nil)
defer db.Close()
docs, err := db.CreateCollection("docs", veclite.CollectionOptions{Dimension: 384})
err = docs.UpsertBatch(points)
hits, err := docs.Search(query, veclite.SearchOptions{Limit: 10, Filter: f})
```

### C# (.NET)

```csharp
using var db = VecLite.Open("app.veclite");
var docs = db.CreateCollection("docs", new CollectionOptions { Dimension = 384 });
docs.UpsertBatch(points);                               // ReadOnlySpan<float> interop
var hits = docs.Search(query, new SearchOptions { Limit = 10 });
```

`SafeHandle`-wrapped native handles; `Span<T>`/pinned-memory pathways for zero-copy vectors.

### WASM (`@veclite/wasm`)

Constraints: no threads by default, no mmap, no file locks, no ONNX.

- Storage backends: in-memory (always), **OPFS** (browser persistent, via JS async shim around a sync core buffer), bytes import/export (`db.serialize(): Uint8Array` / `VecLite.deserialize(bytes)`) for edge/KV persistence.
- Same API surface minus `vacuum` niceties; SIMD via `simd128` (the `simd-wasm` kernels already exist in `vectorizer-core`).
- Positioning: client-side semantic search over small/medium corpora (≤ ~500 k vectors), offline apps, extensions.

## Binding conformance suite

One YAML-driven test corpus (operations + expected results) executed by every binding's CI, guaranteeing behavioral parity: same defaults, same error codes, same filter semantics, same scores (float tolerance 1e-5). A binding is release-blocked until conformance passes on the full platform matrix. (Pattern borrowed from the server's SDK-parity discipline, e.g. the v3.3.0 SDK parity work.)

## Versioning

- Core and all bindings release in lockstep (`veclite 1.2.0` everywhere), matching the Vectorizer SDK convention of tracking the server version.
- FFI ABI is additive-only within a major; `vl_abi_version()` gates loaders.
