# VecLite benchmark harness

A reproducible comparison of VecLite against peer **embedded** vector stores —
sqlite-vec, LanceDB, Chroma — and, optionally, the Vectorizer server (SPEC-015
TST-042, DAG T6.4).

## Run it

```bash
pip install -r bench/requirements.txt        # pinned peer versions
pip install hivellm-veclite                  # or build a release wheel (see below)
python bench/harness.py --vectors 20000 --queries 200 --dim 256 --k 10
```

Flags: `--vectors N --queries Q --dim D --k K --metric cosine|l2 --clusters C
--stores a,b,c --vectorizer-url URL --out results.json`. A store whose package
is not installed is **skipped and reported as such** — never silently dropped,
never fabricated. The numbers that print are the numbers that ran.

Results are written to `bench/results/<timestamp>.json` (hardware profile +
config + per-store numbers + skips) and printed as a markdown table.

### Measure a release build of VecLite

VecLite is measured through its Python wheel, so the wheel must be an
**optimized release build** for a fair comparison against the peers (which ship
release binaries). A debug wheel is ~10× slower and would misrepresent VecLite.
Build and install one:

```bash
maturin build --release --manifest-path crates/veclite-py/Cargo.toml --out target/wheels
pip install --force-reinstall target/wheels/hivellm_veclite-*.whl
```

## Methodology

- **Dataset**: a **clustered** mixture of Gaussians (`--clusters`, default 64),
  representative of real embeddings which live near topic centroids — *not*
  uniform random, the known ANN worst case. Generated by a seeded NumPy RNG, so
  the dataset is a pure function of the flags: rerunning reproduces the recall
  exactly and the latencies within the machine's run-to-run variance.
- **Workload**: build the index once, then run `--queries` nearest-neighbour
  queries at top-`--k`.
- **Metrics**: build time (s); query latency p50 / p95 (ms); recall@k against an
  exact brute-force reference (recall `1.0` = an exact store, `< 1.0` = an
  approximate ANN index); throughput (queries/s, from the mean latency).
- **Fairness**: every store uses its idiomatic embedded API at full precision
  (no quantization), the same dataset, the same queries. Approximate stores
  (VecLite, Chroma) use their default HNSW parameters; exact stores (sqlite-vec,
  and LanceDB without an ANN index) scan — the recall column makes the
  exact-vs-approximate distinction transparent.

## Honesty

Publish losses alongside wins. VecLite's HNSW build is **slower** than the
flat-index peers and slower than Chroma's HNSW; that trade buys the fastest
queries and the single-file format. The report in
[`docs/src/reference/benchmarks.md`](../docs/src/reference/benchmarks.md)
presents the full picture with the hardware disclosed. Disclose your own
hardware profile (the harness records CPU/OS/Python in every result) with any
number you publish.

## The Vectorizer server

The server adapter is gated on `--vectorizer-url` and is a stub to wire to your
instance's `/insert_vectors` + `/search` endpoints — the parity gate
(`crates/veclite/tests/parity.rs`) already has the exact BYO-vector calls. Server
parity (top-10 overlap ≥ 0.99) is measured there and recorded in
[`docs/benchmarks.md`](../docs/benchmarks.md); this harness focuses on the
embedded peers.
