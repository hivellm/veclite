# phase2c: locking/read-only shipped; mmap + HNSW persistence blocked by hnsw_rs (ADR-0003)
**Source**: manual
**Date**: 2026-07-14
**Related Task**: phase2c_mmap-lock-readonly
**Tags**: phase2c, storage, locking, read-only, hnsw, mmap, adr-0003, veclite
phase2c_mmap-lock-readonly delivered single-writer locking (STG-060), read-only open with the WalPending guard (STG-062), and damaged-tail tolerance (STG-003). The two heavier goals in the original scope — mmap-as-primary-store for larger-than-RAM (STG-004) and HNSW-graph persistence with rebuild fallback (STG-063) — were split into phase2f_mmap-hnsw-persistence and documented in ADR-0003, because the pinned `hnsw_rs =0.3.4` blocks both: (1) it holds a full f32 copy of every vector inside the graph in RAM, so mmapping VECTORS segments neither enables larger-than-RAM nor cuts memory (the graph itself doesn't fit); (2) it has no stable graph serialization (the reason for the exact-version pin CORE-030), so the graph can't be persisted byte-stably and is always rebuilt from vectors on open. The unblock is an index-strategy decision (custom/forked HNSW reading from mmap, or flat/IVF), which phase2f must make in its own ADR before implementing.

Test-side gotchas found: (a) any test that CREATES a pager and then re-OPENs the same path in the same process must drop the creator first — the create holds the exclusive lock and a second open now self-conflicts with Locked (fixed tail_grows_across_checkpoints_no_overwrite by adding drop(p)); (b) simulating a crash in tests must use the __test_simulate_crash() hook (marks an AtomicBool so Drop skips checkpoint but still releases the OS lock) — mem::forget(db) leaks the lock and the next open gets Locked, because a real process exit releases the OS lock but an in-process forget does not.
